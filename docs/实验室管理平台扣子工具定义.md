# 实验室管理平台扣子工具定义

> 用途：为扣子空间工作流提供可直接映射的工具定义，统一 `lab_manager` 与扣子之间的输入输出协议。
>
> 版本：v1.0
>
> 适用范围：硬件查询、产品缺口分析、任务检索、任务视频抓取、两段式硬件导入

---

## 一、设计原则

本工具定义遵循以下原则：

1. 统一走平台后端网关，不让扣子直接访问数据库。
2. 工具返回尽量结构化，减少大模型二次猜测。
3. 所有权限先由平台后端过滤，再把可见数据返回给扣子。
4. 写操作必须幂等或可审计，尤其是批量导入。
5. 两段式导入严格拆分为 `validate` 和 `commit` 两个工具。

---

## 二、通用调用规范

### 2.1 网关地址

建议使用平台内部地址前缀：

```text
/api/agent/
```

当前项目本次已先在 `lab_manager` 插件内实现对应接口，默认访问路径为：

```text
/plugins/lab-manager/api/agent/
```

如果后续需要统一成全局 `/api/agent/`，建议再在 NetBox 根路由层增加一层转发或网关映射。

### 2.2 通用请求头

```http
Content-Type: application/json
X-Agent-Source: coze-workflow
X-Agent-Token: {internal_service_token}
X-Request-ID: {request_id}
X-User-ID: {current_user_id}
```

说明：

1. `X-Agent-Token` 用于平台识别调用来源。
2. `X-User-ID` 用于后端按当前用户权限过滤结果。
3. `X-Request-ID` 用于链路追踪和审计。

### 2.3 通用返回结构

```json
{
  "success": true,
  "message": "ok",
  "data": {},
  "meta": {
    "request_id": "req_20260620_001"
  }
}
```

### 2.4 通用错误码

| 错误码 | 含义 |
|---|---|
| `40001` | 参数格式错误 |
| `40002` | 必填字段缺失 |
| `40003` | 枚举值非法 |
| `40101` | 网关鉴权失败 |
| `40301` | 当前用户无权限 |
| `40401` | 数据不存在 |
| `40901` | 数据状态冲突 |
| `40902` | 批次已导入，禁止重复提交 |
| `42201` | 预校验未通过 |
| `50001` | 平台内部异常 |

## 三、工具一：`search_hardware`

### 3.1 用途

用于查询实验室硬件库存，支撑以下场景：

1. 问当前有哪些硬件资源
2. 按类别、型号、状态、保管人、位置搜索
3. 查询库存数量和可用情况

### 3.2 请求定义

```json
{
  "keywords": ["stm32", "开发板"],
  "category": "mcu",
  "status": "idle",
  "approval_status": "approved",
  "custodian": "张三",
  "storage_location": "A柜",
  "limit": 20,
  "offset": 0
}
```

### 3.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `keywords` | `string[]` | 否 | 名称、型号、品牌关键词 |
| `category` | `string` | 否 | `mcu/sensor/power/instrument/tool/wire/module/other` |
| `status` | `string` | 否 | `in_use/idle/repair/scrapped` |
| `approval_status` | `string` | 否 | `pending/approved/rejected` |
| `custodian` | `string` | 否 | 保管人名称 |
| `storage_location` | `string` | 否 | 存放位置模糊匹配 |
| `limit` | `int` | 否 | 默认 `20`，最大建议 `100` |
| `offset` | `int` | 否 | 分页偏移量 |

### 3.4 返回定义

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "total": 2,
    "items": [
      {
        "id": 1,
        "name": "STM32F103C8T6 开发板",
        "category": "mcu",
        "category_label": "单片机/开发板",
        "model_number": "STM32F103C8T6",
        "manufacturer": "ST",
        "quantity": 10,
        "status": "idle",
        "status_label": "闲置",
        "storage_location": "A柜-2层",
        "custodian": "张三",
        "purchase_link": "",
        "approval_status": "approved"
      }
    ]
  }
}
```

### 3.5 工作流提示

1. 当用户只是问“有哪些硬件”时，优先调用此工具。
2. 若结果数量过大，应先总结再列前 N 条。
3. 不要让大模型自行猜库存数量，必须以工具返回为准。

## 四、工具二：`analyze_hardware_gap`

### 4.1 用途

用于把“产品需求清单”与“现有库存”进行比对，输出：

1. 已有硬件
2. 缺失硬件
3. 数量缺口
4. 推荐采购项

### 4.2 请求定义

```json
{
  "project_name": "智能温室监测系统",
  "requirements": [
    {
      "name": "ESP32 开发板",
      "category": "mcu",
      "required_quantity": 2,
      "keywords": ["esp32", "wifi 开发板"]
    },
    {
      "name": "温湿度传感器",
      "category": "sensor",
      "required_quantity": 4,
      "keywords": ["温湿度", "dht", "sht"]
    }
  ]
}
```

### 4.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `project_name` | `string` | 否 | 项目名称 |
| `requirements` | `object[]` | 是 | 需求项数组 |
| `requirements[].name` | `string` | 是 | 需求名称 |
| `requirements[].category` | `string` | 否 | 硬件类别枚举 |
| `requirements[].required_quantity` | `int` | 是 | 所需数量，必须大于 0 |
| `requirements[].keywords` | `string[]` | 否 | 用于库存匹配的关键词 |

### 4.4 返回定义

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "project_name": "智能温室监测系统",
    "matched": [
      {
        "name": "ESP32 开发板",
        "required_quantity": 2,
        "available_quantity": 1,
        "gap_quantity": 1,
        "matched_items": [
          {
            "id": 8,
            "name": "ESP32 DevKit",
            "quantity": 1,
            "status": "idle"
          }
        ]
      }
    ],
    "missing": [
      {
        "name": "温湿度传感器",
        "required_quantity": 4,
        "available_quantity": 0,
        "gap_quantity": 4
      }
    ]
  }
}
```

### 4.5 工作流提示

1. 扣子先抽取需求，再调用此工具。
2. 大模型负责补充“采购建议”和“实现方案”文字，不负责算缺口数量。
3. 缺口数量以工具结果为准。

## 五、工具三：`search_tasks`

### 5.1 用途

用于查询任务列表，支撑：

1. 查询某成员的任务
2. 查询最近完成的任务
3. 按状态、优先级、关键词筛选任务

### 5.2 请求定义

```json
{
  "assigned_to": "李四",
  "created_by": "",
  "status": "completed",
  "priority": "high",
  "keyword": "电机",
  "date_from": "2026-06-01",
  "date_to": "2026-06-20",
  "limit": 20,
  "offset": 0
}
```

### 5.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `assigned_to` | `string` | 否 | 执行人 |
| `created_by` | `string` | 否 | 创建人 |
| `status` | `string` | 否 | `pending/in_progress/completed` |
| `priority` | `string` | 否 | `low/medium/high/urgent` |
| `keyword` | `string` | 否 | 标题或描述关键词 |
| `date_from` | `string` | 否 | 起始日期 |

| `limit` | `int` | 否 | 默认 `20` |

## 七、工具五：`validate_hardware_import`

### 7.1 用途

两段式导入第一阶段，执行预校验，不直接入库。

### 7.2 请求定义

```json
{
  "source_type": "json",
  "import_action": "validate",
  "items": [
    {
      "name": "MPU9250 九轴传感器",
      "category": "sensor",
      "model_number": "MPU9250",
      "manufacturer": "TDK",
      "quantity": 5,
      "status": "idle",
      "storage_location": "B柜-1层",
      "purchase_link": "",
      "remarks": "新采购"
    }
  ]
}
```

### 7.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `source_type` | `string` | 是 | `json/excel/csv/text` |
| `import_action` | `string` | 是 | 固定为 `validate` |
| `items` | `object[]` | 是 | 待导入硬件项 |
| `items[].name` | `string` | 是 | 硬件名称 |
| `items[].category` | `string` | 是 | 合法类别枚举 |
| `items[].quantity` | `int` | 是 | 必须大于 0 |
| `items[].status` | `string` | 否 | 默认 `in_use` 或平台默认值 |

### 7.4 返回定义

```json
{
  "success": true,
  "message": "validated",
  "data": {
    "batch_id": "imp_20260620_001",
    "status": "validated",
    "summary": {
      "total": 3,
      "valid_count": 2,
      "error_count": 1,
      "duplicate_count": 1
    },
    "valid_items": [
      {
        "row_no": 1,
        "name": "MPU9250 九轴传感器",
        "category": "sensor",
        "quantity": 5
      }
    ],
    "duplicate_items": [
      {
        "row_no": 2,
        "name": "L298N 电机驱动模块",
        "matched_hardware_id": 10,
        "reason": "名称+型号命中现有库存"
      }
    ],
    "errors": [
      {
        "row_no": 3,
        "field": "category",
        "code": "40003",
        "message": "类别不合法"
      }
    ]
  }
}
```

### 7.5 工作流提示

1. 预校验通过后，必须向用户展示 `batch_id` 和校验摘要。
2. 只有用户明确确认后，才能调用 `commit_hardware_import`。
3. 发现错误时，应引导用户修改后重新校验。

## 八、工具六：`commit_hardware_import`

### 8.1 用途

两段式导入第二阶段，按 `batch_id` 执行正式入库。

### 8.2 请求定义

```json
{
  "import_action": "commit",
  "batch_id": "imp_20260620_001",
  "confirm": true
}
```

### 8.3 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `import_action` | `string` | 是 | 固定为 `commit` |
| `batch_id` | `string` | 是 | 预校验返回的批次 ID |
| `confirm` | `bool` | 是 | 必须为 `true` |

### 8.4 返回定义

```json
{
  "success": true,
  "message": "imported",
  "data": {
    "batch_id": "imp_20260620_001",
    "status": "imported",
    "summary": {
      "total": 2,
      "success_count": 2,
      "failed_count": 0
    },
    "created_ids": [21, 22]
  }
}
```

### 8.5 状态约束

1. 只有 `validated` 状态的批次才能提交。
2. `imported` 状态批次必须拒绝再次导入，并返回错误码 `40902`。
3. 批次内无有效数据时，必须拒绝提交。

### 8.6 工作流提示

1. 如果接口返回 `40902`，应明确告诉用户该批次已经导入过。
2. 如果返回 `42201`，说明预校验未通过，不能直接入库。

## 九、工作流编排建议

### 9.1 产品分析类工作流

推荐顺序：

1. LLM 抽取项目需求
2. 调用 `analyze_hardware_gap`
3. LLM 生成采购建议和实现方案

### 9.2 任务视频检索工作流

推荐顺序：

1. 调用 `search_tasks`
2. 提取任务 ID
3. 调用 `search_task_videos`
4. LLM 汇总输出

### 9.3 两段式导入工作流

推荐顺序：

1. 解析上传内容
2. 调用 `validate_hardware_import`
3. 向用户展示校验结果
4. 用户确认
5. 调用 `commit_hardware_import`
6. 返回导入结果

## 十、后续实现建议

1. 先在 `lab_manager` 中实现上述 6 个接口。
2. 接口层统一做用户权限过滤和审计日志。
3. 扣子工作流严格按工具返回结构组织答案，不自行补造库存数据。
4. 所有写操作保留 `request_id`、`batch_id`、`user_id` 以便追踪。
