# 实验室管理平台智能体接入方案

> 基于现有 `NetBox + lab_manager` 项目，接入扣子空间工作流，新增一个面向实验室成员和管理员的智能体对话界面。
>
> 文档目标：让系统具备“可对话查询硬件资源、可做产品硬件缺口分析、可读取任务与视频附件、可批量导入硬件资源”的能力。

---

## 一、建设目标

本次新增能力不是单独再做一个 AI 演示页面，而是把 AI 作为实验室管理平台中的一个正式功能模块接入到现有系统中。用户进入平台后，可以在一个“智能体助手”页面直接完成以下事情：

1. 查询当前实验室有哪些硬件资源，按类别、型号、数量、状态、保管人、存放位置进行问答式检索。
2. 描述自己想做的产品或项目，智能体结合当前硬件库存给出：
   - 已有硬件
   - 缺失硬件
   - 推荐采购清单
   - 一版可执行的实现方案
3. 读取实验室成员的任务列表，识别已完成任务，并把任务上传的图片/视频附件抓取出来，供用户查看或汇总。
4. 支持管理员通过“一键批量上传硬件资源”的方式，把表格中的硬件信息批量导入系统，并生成导入结果反馈。

本方案采用“**你的项目作为业务数据底座，扣子空间作为智能体编排层**”的方式建设，而不是把业务逻辑全部放到扣子里。这样可以保证数据仍然归你自己的系统管理，AI 只负责理解需求、调用接口、组织结果。

## 二、现有项目基础

结合当前代码，项目已经具备本次智能体接入的核心数据基础：

### 2.1 已有模型

1. `Hardware`
   - 存储硬件名称、类别、型号、品牌、数量、单价、购买链接、状态、存放位置、保管人、审批状态等信息。
   - 已经能作为“硬件资源查询”和“采购建议”的基础数据源。

2. `Task`
   - 存储任务标题、描述、优先级、状态、创建人、执行人、截止时间、完成说明等信息。
   - 已经能作为“任务列表读取”和“任务完成度分析”的基础数据源。

3. `TaskAttachment`
   - 存储任务附件文件，当前图片和视频统一通过此模型上传。
   - 已经能作为“抓取任务完成时上传的视频”能力的数据入口。

### 2.2 已有业务页面

当前项目中已经存在：

1. 硬件列表、详情、编辑、审批页面
2. 任务列表、详情、评论、上传附件、任务完成页面
3. 我的任务页面
4. 首页统计页面

这意味着本次新增 AI 功能时，不需要重做硬件、任务、附件三套业务，只需要在现有数据和页面之上新增：

1. 智能体入口页面
2. 给扣子空间调用的业务接口
3. AI 编排工作流
4. 权限与审计机制

## 三、总体方案

### 3.1 总体思路

整体采用四层架构：

```text
用户浏览器
    ->
实验室管理平台 Web 页面
    ->
平台后端 AI 网关接口
    ->
扣子空间 Bot / 工作流
    ->
平台业务接口（硬件、任务、附件、批量导入）
```

对应职责如下：

1. 平台前端
   - 新增“智能体助手”页面
   - 承载聊天窗口、消息历史、建议卡片、推荐采购清单、视频结果展示

2. 平台后端
   - 负责用户身份校验
   - 负责把当前登录用户信息传给扣子工作流
   - 负责提供业务 API 给扣子调用
   - 负责拦截越权访问和记录操作日志

3. 扣子空间
   - 负责自然语言理解
   - 负责意图识别、流程编排、调用工具/API、结果整合
   - 负责把复杂问答拆成多个节点执行

4. 业务数据层
   - 仍由你现有的 `Hardware`、`Task`、`TaskAttachment` 和后续新增的导入记录表来承载

### 3.2 为什么推荐这个方案

1. 数据不出核心业务系统
   - 硬件、任务、附件仍由你的系统保存，扣子只做调用和推理。

2. 方便逐步上线
   - 可以先做“硬件查询 + 产品缺口分析”，后做“任务视频汇总”和“批量导入”。

3. 便于控权
   - 普通成员和管理员看到的数据范围可以在本系统接口层控制，不必把权限判断交给提示词。

4. 后续可扩展
   - 后面还能继续加“采购审批建议”“设备借还推荐”“实验报告生成”等能力。

## 四、目标功能拆解

### 4.1 功能一：智能问答查询硬件资源

#### 用户场景

1. “现在实验室有哪些 STM32 开发板？”
2. “有哪些空闲示波器？”
3. “帮我看一下单片机类硬件库存够不够”
4. “张三保管的硬件有哪些？”

#### 实现方式

扣子工作流先识别用户意图为“硬件查询”，然后调用平台的硬件查询 API，按条件检索并返回结构化数据，再由大模型组织为自然语言答复。

#### 建议输出格式

1. 总结语
2. 硬件列表
3. 数量统计
4. 风险提醒

示例：

```text
当前可用的 STM32 相关硬件共 18 件，其中：
- STM32F103C8T6 开发板：10 件，空闲 6 件
- STM32H750 开发板：4 件，空闲 1 件
- ST-Link 下载器：4 件，空闲 2 件

如果你准备做中等复杂度控制类项目，现有库存基本够做 2 到 3 组并行开发。
```

### 4.2 功能二：输入产品目标，AI 分析缺少哪些硬件

#### 用户场景

1. “我想做一个智能温室监测系统，需要哪些硬件？”
2. “我想做一台带屏幕的智能小车，看看实验室还缺什么？”
3. “我要做一个多路传感采集终端，帮我按现有库存出方案”

#### 实现方式

这个功能分成 4 步：

1. 提取项目需求
   - 从用户自然语言中提取产品类型、核心功能、通讯方式、供电方式、是否需要显示、是否需要联网、是否需要执行器等关键信息。

2. 生成标准硬件需求清单
   - 由扣子中的提示词或规则节点，输出“主控、电源、传感器、执行器、通信模块、结构件、调试工具”等分类清单。

3. 对照实验室现有库存
   - 调用平台硬件检索接口，按型号关键词、类别、状态、数量比对。

4. 生成建议结果
   - 输出“已有硬件”“缺少硬件”“建议采购件”“可行实施方案”“预算粗估”。

#### 输出结果建议

应返回结构化卡片：

1. 项目理解
2. 已有资源
3. 缺失资源
4. 推荐采购链接或采购关键词
5. 组装建议
6. 风险项

#### 关键价值

这个能力的本质不是单纯聊天，而是把“实验室资源盘点 + 采购前评估 + 初步方案设计”合成一次对话完成，能直接提高硬件决策效率。

### 4.3 功能三：读取成员任务列表并抓取完成任务中的视频

#### 用户场景

1. “帮我看一下最近 7 天完成了哪些任务，并把视频找出来”
2. “张三已完成任务里有哪些上传了视频？”
3. “把电机调试相关任务的视频都列出来”

#### 实现方式

扣子工作流识别为“任务与视频检索”后，调用以下两类接口：

1. 任务查询接口
   - 根据成员、状态、时间范围、关键词获取任务列表

2. 附件查询接口
   - 针对任务 ID 列表获取所有附件
   - 过滤出视频类型附件

#### 附件识别规则

平台后端不建议仅让大模型猜文件类型，而应在接口层直接给出：

1. `file_type=image|video|other`
2. `file_name`
3. `file_url`
4. `uploaded_by`
5. `created`
6. `task_id`
7. `task_title`

这样扣子只需要负责组织结果，不需要自己判断文件是不是视频。

#### 前端展示建议

结果页建议展示：

1. 任务标题
2. 完成时间
3. 执行人
4. 视频缩略信息
5. “打开附件”按钮

如果后续要做更好体验，可以增加：

1. 视频封面截图
2. 任务视频合辑页
3. 一键导出视频清单

### 4.4 功能四：AI 一键批量上传硬件资源

#### 用户场景

1. 管理员上传一份 Excel，AI 自动识别字段并导入
2. 管理员说“把这份采购表里的硬件全部录入系统”
3. 管理员把表格粘贴到对话框中，让 AI 解析并完成批量导入

#### 推荐实现方式

建议分成两种入口：

1. 页面上传入口
   - 在智能体页面旁边提供“上传 Excel/CSV”按钮
   - 文件先上传到平台后端，再触发扣子工作流进行字段映射分析

2. 对话粘贴入口
   - 用户把表格内容直接粘到聊天框
   - 扣子解析为结构化 JSON
   - 再调用平台批量导入 API

#### 导入流程

1. 上传原始文件或文本
2. 扣子解析字段
3. 形成标准硬件数据
4. 调用“批量预校验接口”
5. 返回预校验结果
6. 用户确认
7. 调用“正式导入接口”
8. 返回导入成功/失败明细

#### 为什么必须做“两段式导入”

如果直接让 AI 一步落库，风险较大：

1. 字段识别可能错误
2. 型号和数量可能被误判
3. 重复导入难追溯

因此建议一定使用：

1. 预校验
2. 用户确认
3. 正式入库

三步机制来降低误操作风险。

## 五、系统架构设计

### 5.1 平台侧新增模块

建议在现有 `lab_manager` 插件中新增以下能力：

1. 智能体页面模块
   - 路径示例：`/plugins/lab-manager/agent/`

2. AI 网关模块
   - 用于向扣子 API 发起请求
   - 隔离 Token、Bot ID、Workflow ID 等配置

3. AI 业务 API 模块
   - 提供给扣子工作流调用的内部 HTTP 接口

4. 导入记录模块
   - 记录每次批量导入的来源、导入人、明细、失败原因

### 5.2 推荐新增的数据对象

建议新增 3 个模型：

#### 1. AgentConversation

用于记录聊天会话：

```python
class AgentConversation(NetBoxModel):
    user = models.ForeignKey(User, on_delete=models.CASCADE)
    session_id = models.CharField(max_length=100, unique=True)
    title = models.CharField(max_length=200, blank=True)
    source = models.CharField(max_length=30, default='coze')
```

#### 2. AgentMessage

用于记录消息历史：

```python
class AgentMessage(NetBoxModel):
    conversation = models.ForeignKey(AgentConversation, on_delete=models.CASCADE, related_name='messages')
    role = models.CharField(max_length=20)  # user / assistant / system
    content = models.TextField()
    raw_payload = models.JSONField(default=dict, blank=True)
```

#### 3. HardwareImportBatch

用于记录批量导入过程：

```python
class HardwareImportBatch(NetBoxModel):
    created_by = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)
    source_type = models.CharField(max_length=20)  # excel / csv / text
    original_filename = models.CharField(max_length=255, blank=True)
    status = models.CharField(max_length=20)  # draft / validated / confirmed / imported / failed
    parsed_payload = models.JSONField(default=list, blank=True)
    result_summary = models.JSONField(default=dict, blank=True)
```

## 六、扣子空间工作流设计

### 6.1 接入方式建议

建议使用：

1. 扣子空间中的一个主 Bot
2. Bot 下挂多个工作流
3. 平台后端统一调用扣子 API

推荐原因：

1. Bot 负责统一对话体验
2. 工作流负责稳定执行结构化流程
3. 平台后端负责安全控制和日志记录

### 6.2 工作流拆分建议

不要把所有能力塞进一个超大工作流，建议拆成 4 条：

#### 工作流 A：硬件资源查询

输入：

1. 用户问题
2. 当前用户 ID
3. 可见数据范围

输出：

1. 查询摘要
2. 硬件清单
3. 统计结果

#### 工作流 B：产品需求缺口分析

输入：

1. 用户产品描述
2. 当前库存快照或检索结果

输出：

1. 产品理解
2. 所需硬件清单
3. 已有硬件
4. 缺失硬件
5. 采购建议
6. 实施方案

#### 工作流 C：任务与视频抓取

输入：

1. 查询条件
2. 当前用户 ID

输出：

1. 匹配任务列表
2. 视频附件列表
3. 汇总说明

#### 工作流 D：硬件批量导入

输入：

1. 文件内容或文本
2. 导入人 ID
3. 导入模式

输出：

1. 解析结构
2. 预校验结果
3. 导入确认结果

### 6.3 工作流节点示意

以“产品需求缺口分析”工作流为例：

```text
开始
  ->
意图确认
  ->
需求抽取
  ->
生成标准硬件需求
  ->
调用平台硬件查询接口
  ->
库存匹配与数量比对
  ->
LLM 生成缺口分析与采购建议
  ->
结束
```

### 6.4 扣子工作流中的工具能力建议

建议至少定义以下 5 个可调用工具：

1. `search_hardware`
   - 查询硬件资源

2. `analyze_hardware_gap`
   - 对照清单做缺口分析

3. `search_tasks`
   - 查询任务列表

4. `search_task_attachments`
   - 获取任务附件并过滤视频

5. `bulk_import_hardware`
   - 批量导入硬件资源

其中第 2 个工具也可以由平台后端完成，避免完全依赖模型自由发挥。

## 七、平台接口设计

### 7.1 接口设计原则

给扣子调用的接口不要直接暴露原始数据库结构，而应提供“适合 AI 使用”的结构化接口，特点是：

1. 字段尽量语义化
2. 输出尽量扁平
3. 权限先过滤再返回
4. 支持分页和数量限制
5. 支持按需返回摘要字段

### 7.2 推荐接口清单

#### 1. 查询硬件资源

`POST /api/agent/hardware/search/`

请求示例：

```json
{
  "keywords": ["stm32", "开发板"],
  "category": "mcu",
  "status": "idle",
  "limit": 20
}
```

返回示例：

```json
{
  "total": 2,
  "items": [
    {
      "id": 1,
      "name": "STM32F103C8T6 开发板",
      "category": "mcu",
      "model_number": "STM32F103C8T6",
      "manufacturer": "ST",
      "quantity": 10,
      "status": "idle",
      "storage_location": "A柜-2层",
      "custodian": "张三",
      "purchase_link": ""
    }
  ]
}
```

#### 2. 产品硬件缺口分析

`POST /api/agent/hardware/gap-analysis/`

说明：

1. 平台后端接收扣子整理后的需求清单
2. 后端执行库存匹配
3. 返回已有、缺失、短缺数量

这样能让结果更加稳定，避免同一问题每次分析结果都漂移。

#### 3. 查询任务列表

`POST /api/agent/tasks/search/`

支持条件：

1. 执行人
2. 创建人
3. 状态
4. 时间范围
5. 关键词

#### 4. 查询任务视频附件

`POST /api/agent/tasks/videos/`

建议后端直接返回：

```json
{
  "tasks": [
    {
      "task_id": 12,
      "task_title": "电机驱动调试",
      "assigned_to": "李四",
      "status": "completed",
      "completed_at": "2026-06-18T10:00:00+08:00",
      "videos": [
        {
          "attachment_id": 23,
          "file_name": "motor_test.mp4",
          "file_url": "/media/task_attachments/motor_test.mp4",
          "uploaded_by": "李四",
          "created": "2026-06-18T09:52:00+08:00"
        }
      ]
    }
  ]
}
```

#### 5. 批量预校验硬件导入

`POST /api/agent/hardware/import/validate/`

#### 6. 确认正式导入

`POST /api/agent/hardware/import/commit/`

## 九、权限与安全设计

### 9.1 权限原则

AI 不能突破现有系统权限，所有权限必须由平台后端控制。

建议权限规则如下：

1. 普通成员
   - 可查询自己可见的硬件
   - 可查看与自己相关的任务
   - 可查看自己有权限访问的任务附件
   - 不可执行正式批量导入

2. 管理员
   - 可查询全部硬件
   - 可查询全部任务
   - 可查看全部任务附件
   - 可执行批量导入和确认入库

### 9.2 安全措施

建议至少做到：

1. 扣子调用平台接口时使用服务端签名或内部 Token
2. 平台对外不直接暴露数据库明细接口
3. 对文件 URL 做登录鉴权或临时签名
4. 对批量导入操作做管理员确认
5. 记录所有 AI 发起的重要操作日志

### 9.3 审计建议

## 十二、关键风险与应对

### 风险一：AI 回答不稳定

原因：

1. 大模型自由生成过多
2. 接口返回结构不稳定

应对：

1. 尽量让平台接口返回结构化结果
2. 缺口分析优先在后端计算
3. 大模型只负责解释与组织语言

### 风险二：越权读取任务和附件

原因：

1. 把权限交给提示词判断

应对：

1. 所有权限在平台接口层处理
2. 扣子只拿到已过滤结果

### 风险三：批量导入误操作

原因：

1. 表格字段识别错误
2. 同一批数据重复导入

应对：

1. 预校验后人工确认
2. 导入批次留痕
3. 增加重复数据检测

### 风险四：视频文件访问性能差

原因：

1. 附件文件较大

应对：

1. 结果页只展示视频列表，不直接全部自动播放
2. 后端返回分页结果
3. 后续可增加缩略图和对象存储

---

## 十三、推荐落地顺序

如果你希望尽快看到效果，建议按下面顺序开发：

1. 先做智能体页面和硬件查询
2. 再做“我想做什么产品”的缺口分析
3. 然后做任务视频检索
4. 最后做批量导入

这样做的原因是：

1. 前两项最容易出效果，最适合先验证价值
2. 任务视频检索依赖附件权限控制，复杂度中等
3. 批量导入涉及写操作和数据风险，适合最后做

## 十四、结论

你的现有项目已经具备接入智能体的很好基础，因为 `Hardware`、`Task`、`TaskAttachment` 三个核心对象已经存在。此次最合理的方案不是重构业务模型，而是在现有插件基础上新增“AI 页面 + AI 网关 + 业务接口 + 扣子工作流”四部分。

按本方案实施后，平台可实现以下结果：

1. 用户能像聊天一样查询实验室硬件资源
2. 用户能输入产品想法并得到基于库存的硬件缺口分析和采购建议
3. 用户能快速定位成员已完成任务中的视频附件
4. 管理员能通过 AI 辅助完成硬件资源的批量导入

这套方案兼顾了可落地性、权限安全性和后续扩展空间，适合作为你当前项目的下一阶段重点功能。
